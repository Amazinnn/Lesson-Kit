# Gate: Question Specificity

Checkpoint questions and MCQs must target concrete knowledge details.

| Failure | Return to | Forbidden repair |
|---|---|---|
| Question asks for vague understanding or summary | `intermediate/first_pass/02_analysis/checkpoint-question-candidate-map.md` | Adding source locator after a vague question |
| Question is too broad for one learning item | `intermediate/first_pass/02_analysis/learning-item-granularity-analysis.md` | Accepting a long-answer checkpoint |
| MCQ tests general impression instead of a detail | `intermediate/first_pass/03_plans/checkpoint-question-final-plan.md` | Making options shorter only |
| Answer includes long explanation | `intermediate/first_pass/03_plans/checkpoint-question-final-plan.md` | Moving the long explanation below the question |
| MCQ tests definition recall or formula recognition instead of scene judgment | `intermediate/first_pass/03_plans/first-pass-structure-plan.md` | Rewriting the stem without changing the question target type |
| MCQ can be answered correctly from the quick_absorption interpretation alone, without returning to the source | `intermediate/first_pass/03_plans/first-pass-structure-plan.md` | Making the source locator more explicit while keeping an easy question |

Good questions refer to a definition condition, variable role, formula condition, algorithm field, code step, diagram relation, signal/state, table cell, or failure boundary.
